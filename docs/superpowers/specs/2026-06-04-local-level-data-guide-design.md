# Design Spec — Illustrated Guide to Creating Local-Level Geographic Datasets

**Date:** 2026-06-04
**Status:** Draft for review
**Author:** Aaron Schroeder (with Claude)

## 1. Summary

Rework the existing "Guide to Creating Combined Local-Level Geographic Datasets"
from a text-heavy, white-paper-style document into a **professional, illustrated
guide** for local-government policy analysts and decision-makers. The guide
teaches a general method — moving data from misaligned source geographies onto
the sub-county geographies that actually matter for local policy (neighborhoods,
civic associations, corridors, service zones) — through one deep worked example:
Arlington County broadband (Ookla) × household income (ACS) → the county's 62
civic associations.

The implementation language shown in the document is **Python**, built around the
`sdc-redistribute` package. The document is structured so an **R** implementation
can be slotted in later (the matching R package is being built in parallel by the
author and is out of scope here).

## 2. Goals

- A visually polished, easy-to-read guide that a non-technical decision-maker can
  follow through prose + figures, and a hands-on analyst can reproduce.
- Authored in **Quarto**, rendering a print-quality **PDF** (primary deliverable),
  an **HTML** version (bonus), and keeping the generated `.tex` as a byproduct.
- Methodologically correct areal interpolation: counts redistributed as extensive
  measures, rates/averages derived from count ratios (extensive vs. intensive).
- Built on the author's own `sdc-redistribute` package as the spine of the method.
- Fully reproducible: every figure in the guide is generated by code in a companion
  pipeline, not hand-made or screenshotted.

## 3. Non-goals

- Building or rewriting the **R** `redistribute` package (author handles in parallel).
- Writing the R code tabs now (stubbed; filled after the R package lands).
- Generalizing to multiple worked examples / jurisdictions beyond Arlington (one
  deep example + a short "apply this to your geography" recipe is sufficient).
- Interactive web-app features. HTML is a bonus render, not an application.

## 4. Audience

Two reader modes, served by one document:

- **Decision-makers / non-technical:** read narrative + figures; skip code.
- **Analysts / technical:** expand folded code snippets; reproduce via companion repo.

## 5. Deliverables

1. A Quarto project under `guide/` rendering PDF + HTML (and keeping `.tex`).
2. A reproducible Python pipeline under `pipeline/` that regenerates all data
   products and figures.
3. The four existing GeoJSON datasets retained under `data/`, plus acquisition
   scripts.
4. Code-generated figures (maps + flowcharts) under `guide/figures/`.
5. A top-level `README.md` describing the project and how to reproduce it.
6. Existing consolidated artifacts (`.tex`/`.md`/`.html`/`.pdf`, PPT graphics)
   moved to `archive/` — preserved, not deleted.

## 6. Architecture & sub-projects

This effort is **Sub-project B (the guide)**. **Sub-project A (the R package
mirroring `sdc-redistribute`)** is built in parallel by the author and is a
dependency only for the future R code tabs, not for completing the guide.

```
Data acquisition (Python) ──► data/*.geojson
        │
        ▼
Redistribution (sdc-redistribute, Python) ──► civic-association measures
        │
        ▼
Figure generation (geopandas/matplotlib/contextily) ──► guide/figures/*
        │
        ▼
Quarto guide (PDF / HTML / .tex)  ◄── folded inline snippets + prose
```

## 8. Methodology (correction from current draft)

Reframe both transformations around **extensive counts + derived ratios**:

- **Income:** redistribute *aggregate household income* and *household count* as two
  extensive counts; derive `mean income = total_income / households`. This replaces
  the current draft's "assume uniform income within block groups, then area-average"
  approach, which double-counts the uniformity assumption and mishandles an intensive
  variable.
- **Broadband:** redistribute *test counts* as extensive; treat *download/upload speed*
  as a test-count-weighted rate rather than a raw area-weighted average.

The guide explicitly teaches **why** intensive variables must be derived from
redistributed counts — the most common areal-interpolation error. `redistribute_direct`'s
`count_cols` + `pct_specs` API maps directly onto this pattern.

**Open verification item:** confirm `sdc-redistribute` output on the Arlington data
reproduces (or sensibly corrects) the numbers narrated in the current draft. Several
figures in the existing draft (e.g. "28 of 62 analyzed", non-reconciling percentages,
per-neighborhood Mbps/income values) appear illustrative rather than computed and must
be **regenerated from the pipeline**, with the narrative rewritten to match real output.

## 9. Visual & design system (professional quality is a first-class requirement)

- **Theme/identity:** a custom Quarto theme — deliberate typography (a clean serif for
  body, a humanist sans for headings/labels), generous whitespace, a defined heading
  hierarchy, and a cohesive **colorblind-safe palette** used consistently across every
  figure and accent. A short style sheet defines fonts, palette hex values, and figure
  conventions so the document reads as one designed artifact.

- **Proposed default identity (approved as the starting point; open-source fonts so PDF
  and HTML render identically):**
  - *Typography:* headings/labels in **Libre Franklin** (a humanist, civic sans);
    body in **Source Serif 4** (readable, professional); code in **JetBrains Mono**.
  - *Core palette:* ink navy `#1B2A4A` (primary text/headers), civic blue `#2166AC`
    (primary accent), teal `#2A7F7F` (secondary), warm amber `#D8732E` (callouts/
    highlights), with neutral grays (`#6B7280`, `#E5E7EB`) for rules and panel fills.
  - *Map palettes (all colorblind-safe):* sequential = **viridis** (or ColorBrewer
    `YlGnBu`); diverging, for ratio/balance maps = ColorBrewer `RdBu` reversed; the
    bivariate choropleth = a 3×3 blue–red bivariate scheme.
  - *Figure standards:* 300 DPI for print, a fixed figure width, captions + alt text,
    and maps carrying a scale bar, north arrow, and consistent legend styling — all
    enforced via one shared matplotlib style module.
- **Flowcharts:** authored as **Mermaid or Graphviz** (Quarto-native; render to both PDF
  and HTML; version-controlled). These replace the PowerPoint exports. The two
  transformation diagrams (`ACS Data Transform.png`, `Ookla transform.png`) and the
  data-flow diagrams are redrawn this way, styled to the palette.
- **Maps:** generated from **Python** — `geopandas` + `matplotlib` + `contextily`
  (basemap) + `mapclassify` (classification schemes), with consistent palette, scale
  bar, north arrow, legend, and titling. The bivariate choropliths use a small reusable
  helper. The 3-panel "source → target" figure is regenerated, not screenshotted.
- **Figure standards:** consistent sizing/DPI for print, captions, alt text, and a
  shared plotting style module so all figures share one look.

## 10. Code & reproducibility model

- **In-document:** short, **folded** (`code-fold: true`) Python snippets at the 2–3
  pivotal steps (CRS standardization, the `redistribute_direct` call, a representative
  map render). Decision-makers skim; analysts expand.
- **Companion repo (this `local_level_data` repo):** the full end-to-end runnable
  pipeline. Python stack:
  - `pygris` — census block & block-group geometries
  - `census` / `cenpy` — ACS table B19013 (and the household-count / aggregate-income
    tables needed for the corrected method)
  - direct S3 parquet read (`s3://ookla-open-data/...`) + bbox filter for Ookla tiles
  - `sdc-redistribute` — the redistribution transforms
  - shared mapping/style helpers
- **R:** code tabs/sections left as clearly-marked stubs to fill once the R package lands.
- Reproducibility target: a documented command sequence regenerates every dataset and
  every figure used in the guide.

## 11. Repo organization

```
local_level_data/
  guide/                 # Quarto source (.qmd), _quarto.yml, refs.bib, theme/style
    figures/             #   code-generated maps + rendered flowcharts

  R/                     # stub for the future R implementation
  docs/superpowers/specs/# this spec and future design docs

```

The current `guide_consolidated.*` files, `images/` PPT exports, and loose top-level
`*.R` scripts move into `archive/` (preserved for reference), with the still-useful
acquisition logic (e.g. `download_ookla.R`, `code1.R`) ported into the Python pipeline
or kept in `archive/` for reference.

## 12. Dependencies

- **Authoring:** Quarto (+ a TeX distribution for PDF), Jupyter/Python kernel.
- **Python:** `geopandas`, `pandas`, `pyarrow`, `pygris`, `census`/`cenpy`, `contextily`,
  `mapclassify`, `matplotlib`, `sdc-redistribute`.
- **Diagrams:** Mermaid and/or Graphviz (Quarto-managed).
- **Fonts:** the chosen serif/sans pair must be available to the PDF and HTML renders.

## 13. Risks & open questions

- **Number provenance:** existing narrative figures may not match recomputed values;
  the narrative will be rewritten to match the pipeline output (could shift some
  "findings"). Acceptable — correctness over preserving old prose.
- **Ookla speed as a rate:** confirm the count-weighted derivation matches the intended
  interpretation of "average speed per civic association."
- **R/Python parity:** when the R package arrives, confirm it reproduces the Python
  results so R and Python sections agree.
- **PDF rendering of Mermaid:** verify Mermaid→PDF path works in the chosen Quarto
  setup; fall back to Graphviz/`dot` (or TikZ for the hero diagram) if needed.
- **Font availability** across PDF/HTML render environments.

## 14. Success criteria

- PDF and HTML render cleanly from one Quarto source; `.tex` is retained.
- Every figure is regenerated by pipeline code; no PowerPoint screenshots remain in the
  guide body.
- The redistribution uses `sdc-redistribute` with the extensive/intensive method, and
  the guide explains that distinction.
- A reader can reproduce the full pipeline from the companion repo via documented steps.
- The document looks like a single, professionally designed artifact (consistent theme,
  typography, palette, figure style).
- Old artifacts are preserved in `archive/`, not lost.
