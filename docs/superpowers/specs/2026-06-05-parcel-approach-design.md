# Design Spec — Parcel-Based Redistribution (Second Approach)

**Date:** 2026-06-05
**Status:** Draft for review
**Author:** Aaron Schroeder (with Claude)

## 1. Summary

Add a **second redistribution approach** to the guide: parcel-based dasymetric
redistribution using the existing `sdc_redistribute.redistribute_parcels`, shown
as a comparison against the area-weighted method that the guide already
demonstrates. The point: area-weighting spreads a block group's income uniformly
over its whole area — including parks, roads, and parking lots — while
parcel-weighting places it where housing units actually are. Demonstrated on the
**income** path; broadband (from Ookla tiles) is unchanged.

No changes to `sdc-redistribute`; this uses the existing parcel function. A
later, separate effort may add PUMS household-size weighting to the package.

## 2. Goals

- Acquire Arlington parcel data (the MHUD layer) and redistribute block-group
  income to civic associations *through unit-weighted parcels*.
- Compare the parcel-weighted and area-weighted income estimates per civic
  association, with strong supporting figures.
- Add a guide chapter that teaches dasymetric parcel weighting, shows it, and
  honestly states its limitation (equal weight per *unit*, not yet adjusted for
  household size).

## 3. Non-goals

- Modifying `sdc-redistribute` (the PUMS-adjusted `redistribute_parcel_pums_adj`
  port is deferred to a separate effort).
- Applying parcels to the broadband path.
- Parcel-weighting any measure other than income in this iteration.

## 4. Data

**Arlington MHUD polygons** — `od_MHUD_Polygons` FeatureServer
(`https://arlgis.arlingtonva.us/arcgis/rest/services/Open_Data/od_MHUD_Polygons/FeatureServer/0`).
Verified: **34,340** polygon features; relevant fields `Total_Units` (int),
`Unit_Type` (e.g. `SFD` single-family detached; multifamily types differ),
`Property_Class`, `Full_Block` (census block GEOID). ArcGIS paginates (~2,000
features/request), so acquisition must page through with `resultOffset`.

Output: `data/va013_parcels.geojson` — parcel **centroids** in EPSG:4326 with
`Total_Units` and `Unit_Type` retained.

## 5. Method

**Unit-weighted parcel dasymetric redistribution of income:**

1. Load parcel centroids; **explode** each into `Total_Units` identical points
   (a 300-unit building becomes 300 points; a single-family parcel stays 1).
   Drop parcels with `Total_Units <= 0`.
2. Run `redistribute_parcels` with source = block-group income counts
   (`agg_income`, `households`), the exploded parcel points, and target = civic
   associations. The function assigns each point to its source block group,
   splits the block group's counts evenly across its points, and re-aggregates
   the points falling within each civic association.
3. Derive `mean_income = agg_income / households` per civic association — exactly
   as the area method does (intensive measure from redistributed counts).

Output: `data/civic_income_parcels.csv` (geoid, agg_income, households,
mean_income).

**Why this is more realistic:** allocation is concentrated where housing units
exist, not smeared uniformly across each block group's polygon. **Limitation
(stated in the chapter):** every *unit* gets equal weight, so it does not yet
account for single-family vs multifamily household-size differences — that
refinement is the future PUMS-weighted method.

## 6. Comparison

A compare step joins the area-weighted mean income (from `data/civic_income.csv`,
the existing output) and the parcel-weighted mean income, producing
`data/civic_income_comparison.geojson` per civic association with:
`mean_income_area`, `mean_income_parcel`, `diff = parcel - area`, and
`pct_diff`. Validation: parcel-method household totals preserved within
tolerance vs the ACS source (parcels falling outside all civic associations are
reported, not silently dropped).

## 7. Figures (a strong visual set for the chapter)

Generated by `pipeline/figures.py` via the existing style/`maps` helpers, into
`figures/`:

1. **`map_parcels.png`** — the MHUD parcels as points over Arlington, **sized by
   `Total_Units`** and colored single-family vs multifamily. Shows the dasymetric
   basis: where (and how densely) housing units actually sit. This is the
   chapter's hero figure.
2. **`map_income_parcels.png`** — mean household income by civic association via
   the *parcel* method (same sequential style as the area income choropleth, for
   direct visual comparison).
3. **`map_income_diff.png`** — difference map (`parcel − area`) on a diverging
   palette (RdBu): which associations gain/lose income under parcel placement.
4. **`scatter_area_vs_parcel.png`** — area vs parcel mean income per association,
   with a 1:1 reference line, quantifying how much the two methods diverge.

A new `maps.points_map(...)` helper supports figure 1 (sized/colored point map);
figures 2–4 reuse `choropleth`/`scatter`.

## 8. Guide chapter

New chapter **"A Second Approach: Parcel-Based Redistribution"**, ordered in
`guide/_quarto.yml` immediately after `05-results.qmd` (no file renumbering —
Quarto orders by the `chapters:` list). File: `guide/parcel-approach.qmd`.

Content (~700–900 words):
- The idea: dasymetric redistribution; area-weighting's uniform-distribution
  weakness; parcels as a "where the housing is" mask.
- The data (MHUD parcels) and the unit-weighting (explode by `Total_Units`).
- A folded, static Python snippet showing the `redistribute_parcels` call.
- Embed figures 1–4; report the biggest association-level shifts using the real
  comparison numbers (to be filled from the computed output, not invented).
- Honest limitation + forward pointer to PUMS household-size weighting.

The new pipeline modules appear in the code appendix automatically (it is
generated from `pipeline/*.py` by `scripts/build_code_appendix.py`, whose
`MODULES` list will include the two new modules).

## 10. Dependencies & reproducibility

- Acquisition adds a paged HTTP fetch (stdlib/`requests` via geopandas/`pyarrow`
  already present; use `urllib`/`requests`-free `geopandas.read_file` against the
  ArcGIS query URL per page, or `urllib` + JSON). No new hard dependency expected.
- `pipeline.run` gains the parcel acquisition + redistribution + compare steps;
  `pipeline.build_figures` gains the 4 figures; `quarto render` picks up the new
  chapter. Full reproduction stays: `pipeline.run` → `build_figures` →
  `quarto render`.

## 11. Tests

- `_explode_by_units` (replicate parcels by `Total_Units`, drop ≤0) — deterministic unit test.
- Parcel redistribution on a small synthetic fixture (source polygon + a few
  unit-weighted points + two targets) asserting the count split matches the
  unit weights and totals are preserved — deterministic.
- `compare` join + `diff`/`pct_diff` arithmetic — deterministic unit test.
- `points_map` returns a Figure (smoke); the 4 figures get file-exists smoke
  tests in the existing pattern.
- Acquisition: a `network`-marked smoke test (FeatureServer returns features with
  `Total_Units`/`Unit_Type`).

## 12. Risks

- **Parcels outside civic associations**: some parcels (or block-group income)
  may fall outside the 62 association polygons and lose value; the compare step
  reports any household-total shortfall rather than hiding it.
- **34k parcels exploded by units** could be hundreds of thousands of points —
  watch the `overlay`/`sjoin` cost; acceptable at county scale but note runtime.
- **MHUD `Total_Units` semantics**: confirm it is housing units (not buildings);
  the sample row (`SFD`, `Total_Units: 1`) is consistent. Multifamily rows should
  carry their unit counts — verified at acquisition.
- **Figure 1 density**: 34k points can over-plot; size-by-units + alpha + drawing
  multifamily on top keeps it legible (visual review in the plan).

## 13. Success criteria

- `pipeline.run` produces the parcel income + comparison outputs; household
  totals preserved within tolerance (shortfall reported).
- Four parcel figures generated and visually verified.
- New chapter renders in both PDF and HTML with real comparison numbers and all
  figures resolving; the site republishes cleanly.
- The narrative states the unit-equal-weight limitation and points to the future
  PUMS-weighted method.
